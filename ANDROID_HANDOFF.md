## 1. 项目目标与已经确定的产品决策

目标是在现有 MoePlay 桌面项目上增加 Android 手机版本，并支持竖屏、横屏自由切换，同时尽量共享前端页面、数据模型、SQLite、网络请求和 Rust 业务逻辑。

### 1.1 Android 首版定位

Android 首版是“媒体伴侣版”，不是 Windows 游戏运行器。

保留：

- 游戏库浏览。
- 游戏详情、标签、评分、笔记、收藏和资料编辑。
- 番剧、漫画、继续观看、资源发现。
- 来源管理、下载和任务状态。
- 移动端设置。
- 后续 WebDAV 双端同步。

Android 隐藏或禁用：

- 启动 Windows 游戏。
- Steam/Epic 本机扫描和导入。
- 模拟器导入。
- 存档目录扫描和文件系统监控。
- 托盘、自启动、大屏模式、桌面窗口模式。
- 桌面更新器和桌面诊断。
- 外部播放器和依赖桌面路径的操作。

不应在 Android 上展示明知无法使用的桌面按钮。游戏详情页应明确显示“此设备仅支持游戏资料管理”。

### 1.2 屏幕方向

用户方向模式固定为：

```ts
type OrientationMode = "auto" | "portrait" | "landscape";
```

- `auto`：跟随设备与系统旋转偏好。
- `portrait`：锁定为传感器竖屏。
- `landscape`：锁定为传感器横屏。
- 默认开启“视频全屏时自动横屏”。
- 进入视频全屏前记录用户模式，退出全屏、播放失败、页面离开或回到前台时恢复。
- 方向偏好是设备设置，不参与跨设备同步。

### 1.3 必须保持的边界

- Windows 版现有交互和大屏模式不能被移动端改造破坏。
- 平台差异通过统一 capability 层表达，不要继续在页面中大量散落 `if Android`。
- Android 不申请广泛存储权限，文件导入/导出后续使用系统 Document Picker/content URI。
- 不同步登录密钥、WebDAV 密码、Windows 本地路径、下载文件、缓存、日志和方向偏好。

Rust 已提供并注册 `get_platform_capabilities`。前端启动时读取并缓存能力，设置 `data-platform`，并使用 `MOBILE_ALLOWED_VIEWS` 与 `isViewSupportedOnPlatform` 过滤 Android 路由/导航。

## P1-B：完成 Android 文件选择与 content URI 适配

现有项目部分桌面文件选择基于路径/rfd。Android 必须：

- 使用 Android Document Picker。
- 接受 `content://` URI。
- 不假设 URI 可转为普通文件系统路径。
- 需要长期访问时申请并保存 persistable URI permission。
- 导出时使用 CreateDocument/目录选择，不申请管理全部文件权限。

优先排查所有“选择目录、导入本地文件、选择封面、导入来源、导出备份”路径。不能用的入口先通过 capability 隐藏，随后逐个移动化。

## P1-C：实现完整 WebDAV 双端同步

建议模块拆分：

1. `sync_export`：从 SQLite 生成本设备 `SyncEnvelopeV1`。
2. `sync_webdav`：列出 `MoePlaySync/v1/devices/*/snapshot.json`，下载并校验。
3. `sync_merge`：使用现有 LWW/tombstone 合并。
4. `sync_apply`：单一 SQLite transaction 写入，失败整体回滚。
5. `sync_publish`：上传本设备新 snapshot，使用临时文件/原子替换能力时优先采用。
6. `sync_scheduler`：启动后、恢复前台、手动触发，带最小间隔与重试退避。
7. `sync_preview`：首次同步或 destructiveChange 较大时要求用户确认。

必须为每种实体定义稳定 ID、`updatedAt` 和来源 `deviceId`。同步前先明确当前数据库 schema，不要直接以 `unknown[]` 粗暴序列化所有表。

建议优先覆盖：

- 游戏元数据、标签、评分、笔记、合集。
- 番剧收藏和集数进度。
- 漫画收藏、章节和页码进度。
- 活动记录。
- 可跨设备的外观设置。

明确排除：

- Windows 可执行文件路径和本地安装状态。
- 监控目录、模拟器路径。
- 下载文件、缓存和日志。
- Token、密码、方向偏好。

## 7. 已知高风险点清单

1. **所有改动未提交**：最容易因 reset/clean 丢失。
2. **Android 工程尚未生成**：目前大量原生代码只经过静态推理，没有编译证明。
3. **方向插件 API**：Kotlin package、command 注册、事件权限和 Activity lifecycle 可能需要按 Tauri 生成结果调整。
4. **Android Keyring 初始化**：必须验证 `ndk-context` 以及真机加解密。
5. **返回键异步处理**：`layeredBack` 和 Escape/原生 back 的时序需复核。
6. **Activity 重建恢复**：尚未完成播放/漫画/筛选等状态快照。
7. **Android 文件 URI**：不能把 `content://` 当成普通路径。
8. **WebDAV 目前只有合并模型**：还没有数据库导出、远端传输、事务应用和调度。
9. **CI 从未运行 Android job**：SDK/NDK 版本、CLI 参数和签名脚本均需实际 runner 验证。
10. **Release race**：Windows 与 Android job 对同一个 GitHub Release 的发布时序可能冲突。
11. **Windows 回归**：平台依赖重排后必须在 Windows 上进行完整 Rust 和 Tauri build。
12. **移动页面覆盖不足**：应用壳已完成，但复杂业务页仍可能存在固定宽度、hover 和桌面路径假设。
