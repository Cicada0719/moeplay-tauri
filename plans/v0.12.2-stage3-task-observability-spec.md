# v0.12.2 Stage 3 — Task Producers & Observability SPEC

## Objective

Turn the existing Task Center into an operational control plane. Every long-running or externally dependent operation must be created by backend-owned typed jobs, emit redacted progress/events, and survive/recover correctly across process restarts.

## Foundation API

`task_queue` owns a versioned `JobOperation` enum and must expose:
- `enqueue_operation(title, operation, idempotency_key)`
- `mark_running`, `mark_succeeded`, `mark_failed`
- `append_event(job_id, level, code, message, progress?)`
- `list_events(job_id, after_sequence?, limit)`

`JobOperation` variants:
- `Import { source, reference_id }`
- `Scrape { game_id, provider_id? }`
- `ProviderVerify { media_type, provider_id }`
- `Backup { scope }`
- `Restore { snapshot_id }`
- `DiagnosticsExport`
- `UpdateCheck`

No secrets, raw URLs with userinfo, request headers, prompt/response bodies, or arbitrary command arguments may be serialized in operation payloads or events.

## Persistence

- Schema v6 creates `background_job_events(job_id, sequence, level, code, message, progress, created_at)`.
- Unique `(job_id, sequence)`; index `(job_id, sequence DESC)`.
- Keep at most 200 events per job.
- Purge events for terminal jobs older than 30 days.
- Event sanitizer redacts URLs with credentials, authorization/token/api key values and long AI-like payloads.

## Producer Policy

- Existing synchronous commands retain backward compatibility.
- Add typed async job commands for import/scrape/provider verification and backup/restore/diagnostics/update; their result is a `TaskCenterJob`/job id.
- Workers use fixed backend dispatch only; frontend cannot submit arbitrary operation payloads.
- Worker transitions are monotonic; late completion never overwrites cancelled.
- Pause/resume only downloads. Retry action appears only for producer kinds that have deterministic safe retry semantics.

## Frontend

- Task Center supports a job details drawer with event timeline, redacted error code, source/reference metadata, and contextual action buttons.
- Details fetch is paginated/keyset by event sequence; refresh does not clear current events.
- A global non-intrusive badge shows active/failed task counts.

## Validation

- Rust migration/repository/task queue tests: sequence ordering, retention, cancellation race, sanitation.
- Command contract tests for each typed job starter.
- Frontend store/UI tests: empty/loading/error/events pagination, no sensitive strings rendered.
- Playwright: open details, filter events, verify redaction, action focus returns.
