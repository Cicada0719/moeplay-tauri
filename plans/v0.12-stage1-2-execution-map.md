# Stage 1–2 Multi-Agent Execution Map

## Agent ownership

| Agent | Package | Exclusive write set |
|---|---|---|
| A | S1-A Rust gate closure | `src-tauri/src/lib.rs` |
| B | S1-B frontend validation | `scripts/test-unit.mjs`, `vite.config.ts`, `vitest.config.ts`, `svelte.config.js`, related script tests |
| C | S1-C updater/release readiness | `.github/workflows/{ci,release}.yml`, updater/release verification scripts/tests |
| D | S2-A backend task control plane | `src-tauri/src/task_queue.rs`, `src-tauri/src/repositories/jobs.rs`, `src-tauri/src/commands/tasks.rs`, Rust task tests |
| E | S2-B frontend jobs domain/store | `src/lib/features/jobs/{contracts,api,store}.ts`, store tests |

## Integration owned by coordinator

- S2-C global UI/navigation.
- S2-D DownloadPage compatibility.
- command registration/capability regeneration if new Tauri commands are introduced.
- final test runs and cross-package conflict resolution.

## Rules

- Agents are not alone in the codebase; never revert other changes.
- No agent may edit files outside its exclusive write set without requesting coordinator approval.
- Preserve backward compatibility unless SPEC explicitly allows a versioned command.
- Every agent must list changed files and tests run.
