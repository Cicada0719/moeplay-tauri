# 0.11.9 全能型软件内部 Loop 与记录仪表盘 SPEC

## Summary

目标是把 MoePlay 从“游戏库 + 番剧 + 漫画 + 工具集合”推进成一个本地优先的全能型软件。核心不是继续堆开发者页面，而是在开发侧建立一个可持续运行的 loop：发现资源、接入来源、统一模型、消费播放、记录状态、修复失败、发布更新，再回到下一轮发现。

产品侧不暴露“中枢”或开发控制台。本轮把原先的中枢入口改成用户可理解的“记录”仪表盘：展示游戏游玩时长、最近会话、趋势和继续入口；开发 loop 只沉淀在 SPEC、测试、发布和源接入队列里。

## GitHub 参考生态

- [Predidit/KazumiRules](https://github.com/Predidit/KazumiRules)：MIT；持续更新的 Kazumi 番剧规则仓库，适合作为番剧规则目录、健康检查和导入格式参考。
- [Predidit/Kazumi](https://github.com/Predidit/Kazumi)：GPLv3；只参考反爬验证、网页播放和规则执行思路，不复制实现。
- [tachiyomiorg/extensions](https://github.com/tachiyomiorg/extensions)：Apache-2.0；漫画源扩展生态参考，重点学习“源定义、能力声明、分类/搜索/章节/图片”接口拆分。
- [mihonapp/mihon](https://github.com/mihonapp/mihon)：Apache-2.0；漫画阅读器本体参考，重点学习本地库、阅读历史、下载队列和扩展管理。
- [yuzono/tachiyomi-extensions](https://github.com/yuzono/tachiyomi-extensions)：Apache-2.0；活跃的 Mihon/Tachiyomi fork，当前未发现标准 `repo` 索引分支，适合作为模块组织和源覆盖参考。
- [aniyomiorg/aniyomi](https://github.com/aniyomiorg/aniyomi)：Apache-2.0；动漫 + 漫画统一应用参考，重点学习媒体类型统一和扩展安装体验。
- [aniyomiorg/aniyomi-extensions](https://github.com/aniyomiorg/aniyomi-extensions)：Apache-2.0；动漫/漫画扩展目录参考，适合指导 MoePlay 的多源适配接口。
- [KotatsuApp/kotatsu-parsers](https://github.com/KotatsuApp/kotatsu-parsers)：GPL-3.0；约 1256 个漫画 parser，只做站点覆盖、分类和 parser contract 参考，不直接复制或链接实现代码。

已确认 Tachiyomi/Mihon 与 Aniyomi 官方扩展仓库都采用 `src/<lang>/<source>` 模块化组织，并通过 `repo/index.min.json` 暴露远程索引；部分 fork 只有源码分支，没有标准索引。MoePlay 优先接入索引字段和能力声明，而不是直接运行 Android APK 扩展。Kotatsu 的覆盖面很大，但许可证边界更严格，只进入“研究契约/站点清单”通道。

## Product Loop

1. **发现**：从本地游戏库、番剧规则仓库、漫画源、Bangumi、VNDB、用户历史中发现候选资源。
2. **接入**：把候选来源归一成 source adapter，声明能力：搜索、详情、章节/线路、播放/图片、下载、验证。
3. **归一**：把游戏、番剧、漫画统一映射为 Resource，来源映射为 Source，用户动作映射为 Action。
4. **消费**：进入播放、阅读、启动游戏、下载、外部打开等可执行动作。
5. **沉淀**：写入继续进度、收藏、失败类型、源健康度、用户偏好。
6. **修复**：失败时进入可操作面板：重试、换源、验证、外部打开、报告规则、加入修复队列。
7. **发布**：把通过验证的修复进入 SPEC、测试、PR、Release、自动更新。
8. **再发现**：健康度和历史成功率反哺搜索排序、源推荐和下一轮开发优先级。

## Implementation SPEC

### SPEC-1：记录仪表盘产品入口

- Dock 使用“记录”入口，和游戏库、番剧、漫画并列。
- 新增 `PlayRecordsDashboard.svelte`：展示总游玩时长、会话数、Top 游戏、最近会话、14 天趋势和月度节奏。
- 记录页优先读取 `get_playtime_summary`，普通浏览器预览环境回落到本地游戏库计算。
- 旧 `#loop` hash 兼容跳转到 `records`，但产品界面不再出现“中枢”文案。
- 页面必须适配桌面和窄屏，不使用营销式 hero，不遮挡底部 Dock。

### SPEC-2：多源 Adapter 设计

- 定义统一 Source Adapter 草案：`mediaType`、`capabilities`、`search`、`detail`、`episodes/chapters`、`play/pages`、`download`、`verify`。
- 番剧优先兼容 KazumiRules；漫画优先参考 Tachiyomi/Mihon 的 source extension 模型。
- Source manifest 必须标记 `ecosystem`、`adoptionStrategy`、`licenseRisk`、`indexUrl`，区分“可同步规则”、“可导入索引”、“只能参考契约”和“需手写适配”。
- 第一批漫画/动漫生态入口：KazumiRules、PicACG、Tachiyomi/Mihon、Yuzono、Aniyomi、Kotatsu、CloudStream、外部视频网站。
- `useWebview`、`antiCrawlerConfig`、`adBlocker`、POST 搜索、URL join、验证码验证继续沿用 0.11.8 基础。

### SPEC-2A：远程索引导入器草案

- 先实现只读导入：读取 Tachiyomi/AnsiYomi `index.min.json`，归一为 MoePlay `SourceAdapterManifest`，不下载 APK、不执行扩展。
- 解析字段：`name`、`pkg`、`apk`、`lang`、`version`、`nsfw`、`sources[].name`、`sources[].baseUrl`、`sources[].hasCloudflare`。
- UI 只展示“源可发现/需外部扩展运行时/许可证低风险”，避免误导为已经可直接播放或阅读。
- 只有存在标准 `indexUrl` 的 manifest 进入自动导入队列；Yuzono、Kotatsu 等先进入“参考契约”。
- 下一阶段再评估是否接 Suwayomi/Mihon-compatible runtime 或独立 JS/Rust adapter。

### SPEC-2B：远程索引缓存与候选源目录

- 新增只读缓存表或前端缓存：记录 `repositoryId`、`fetchedAt`、`sourceCount`、`languages`、`cloudflareCount`、`nsfwCount`、`withBaseUrlCount`。
- Source Center 展示候选源目录：按媒体类型、语言、是否 Cloudflare、是否 NSFW、是否有 baseUrl 过滤。
- 候选源状态必须分层：`discoverable`（索引可见）、`requiresRuntime`（需扩展运行时）、`nativeAdapterPlanned`（可手写原生适配）、`unsupported`（许可证或技术边界暂不支持）。
- 缓存失败不影响本地已有游戏/番剧/漫画功能；失败进入任务中心，显示仓库名、失败类型和重试按钮。
- 仍不执行 APK、不绕过站点限制、不绕过付费或加密内容。

当前实现状态：已完成 `ExtensionSourceCandidate`、候选源状态分层、过滤纯函数、缓存摘要纯函数、候选目录 view model（汇总、排序、筛选选项、UI 行）；尚未接真实网络拉取、持久化缓存和任务中心重试。

### SPEC-2C：外部运行时连接器草案

- 参考 [Suwayomi/Suwayomi-Server](https://github.com/Suwayomi/Suwayomi-Server)：它是 Mihon/Tachiyomi 兼容桌面/server 运行时，可安装和执行扩展，默认本地服务端口为 4567。
- GitHub code search 已确认 Suwayomi 暴露 `POST /graphql`、`WS /graphql` 和 `GET /graphql` playground，可作为 MoePlay 外部连接器的第一版探测/只读查询入口。
- 参考 [recloudstream/cloudstream](https://github.com/recloudstream/cloudstream) 与 [recloudstream/cloudstream-extensions](https://github.com/recloudstream/cloudstream-extensions)：偏视频 / 动漫插件生态，GPL-3.0，仅纳入外部运行时与网页兜底设计参考，不直接并入核心实现。
- MoePlay 不内嵌 Suwayomi 代码；先设计外部连接器：检测本地服务、展示连接状态、只读获取已安装源列表。
- 许可证边界：Suwayomi-Server 为 MPL-2.0，适合作为外部进程/服务集成对象，不直接复制实现。
- 连接器失败时回落到远程索引候选源目录，不影响 PicACG、KazumiRules 和本地游戏库。

当前实现状态：已完成 `suwayomiConnector` 纯模型、默认 `127.0.0.1:4567/graphql` 端点构建、`online/authRequired/offline/schemaMismatch` 探测分层、本地探测超时包装、只读 `sources(first)` 和 `extensions(condition: isInstalled)` GraphQL 查询、运行时 snapshot 聚合、Suwayomi 源到候选目录的映射，以及可持久化的本地地址 / 端口 / Token 配置草案。尚未接 Tauri 权限边界、Token 安全存储、产品化源目录页面和运行时阅读查询。

### SPEC-3：源扩展队列

- 番剧：KazumiRules 全量同步、健康度排序、反爬验证、网页播放兜底。
- 漫画：从单一 PicACG store 演进到多源模式，先补“源声明 + 源列表 + 外部规则目录”基础。
- 视频网站：把可网页播放、可提取、仅外部打开三类能力明确区分。
- 下载：番剧 m3u8、漫画章节图片、本地游戏安装包统一进入任务中心。

### SPEC-4：任务与修复中心

- 将规则安装、源验证、下载、刮削、自动更新、播放失败修复统一显示为 Task。
- Task 需要状态、进度、失败类型、可恢复动作和日志入口。
- 先前端聚合展示，后续再抽后端统一任务表。

### SPEC-5：验证与发布 loop

- 每个 SPEC 完成后跑 `git diff --check`、`npm run check`、相关单元测试。
- 涉及播放器、漫画阅读器、下载和更新时补 Playwright 或 Tauri 手动验收。
- 通过后创建 Draft PR；合并后再发 tag/release 和自动更新产物。
